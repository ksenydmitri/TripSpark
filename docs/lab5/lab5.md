# Отчет по инспекции технического долга проекта TripSpark

## 1. Введение и концепция обеспечения качества

В рамках данной работы была проведена инспекция текущего состояния проекта TripSpark с целью выявления технического долга на этапе прототипирования. Согласно принципам гибких методологий (Agile, XP), качество и управление техническим долгом должны быть интегрированы в процесс разработки на самых ранних этапах, чтобы предотвратить экспоненциальный рост стоимости изменений в будущем.

**Цели инспекции:**
1.  **Выявление технического долга:** Идентификация проблем в архитектуре, коде, документации и инфраструктуре проекта, которые могут замедлить дальнейшую разработку.
2.  **Анализ и классификация:** Группировка выявленных проблем по типам (архитектурный, кодовый, тестовый, документационный).
3.  **Разработка плана мероприятий:** Создание структурированного плана по устранению долга с приоритизацией задач.
4.  **Оценка и сравнение:** Сопоставление объема технического долга с объемом еще не реализованного функционала для принятия обоснованных управленческих решений.
5.  **Корректировка плана разработки:** Формулирование рекомендаций по изменению текущего плана работ на основе проведенного анализа.

## 2. Глубокий анализ "Технического долга"

### 2.1. Анализ признаков технического долга

В ходе инспекции были выявлены следующие признаки технического долга в проекте TripSpark, сгруппированные по категориям:

| Категория | Признак | Описание и последствия |
| :--- | :--- | :--- |
| **Архитектура** | Размытие граней Clean Architecture | Документ требований декларирует Clean Architecture, но на текущем этапе (прототип) границы между `data`, `domain` и `presentation` слоями, вероятно, не соблюдены строго. Это приведет к высокому зацеплению (High Coupling) и усложнит тестирование и добавление новых фич (например, бронирования). |
| **Архитектура** | Отсутствие единого источника данных | В требованиях указана работа в оффлайн-режиме, но нет четкой стратегии синхронизации между локальной базой данных (Room) и потенциальным будущим серверным API (Retrofit). Отсутствие четкого слоя `Repository` приведет к дублированию логики и ошибкам синхронизации. |
| **Код** | "Жесткое" кодирование данных | Рекомендации направлений и карта интересов на этапе прототипа, скорее всего, хранятся в виде статических данных или захардкожены в коде UI. Это делает приложение негибким и не поддающимся масштабированию. |
| **Код** | Отсутствие модульности | Проект не разделен на Gradle-модули (`:core`, `:feature:recommendations`, `:feature:wishlist`, `:data`). Это увеличивает время сборки и не позволяет переиспользовать компоненты в других проектах (или потенциальных модулях в будущем). |
| **Тестирование** | Полное отсутствие тестов | В проекте нет ни Unit-тестов (для бизнес-логики), ни интеграционных, ни UI-тестов. Это является критическим техническим долгом, так как делает невозможным регрессионное тестирование при добавлении новых функций или рефакторинге. |
| **Документация** | Неполная архитектурная документация | Существует только документ с требованиями (`requirements_document_rus.md`). Отсутствуют `README.md` с инструкцией по сборке, диаграммы архитектуры (ADL), что затруднит онбординг новых разработчиков. |
| **Процессы** | Отсутствие CI/CD | Нет автоматизированной сборки и проверки (линтеры, компиляция) при каждом пуше в репозиторий. Это увеличивает риск попадания "сырых" ошибок в основную ветку. |
| **Процессы** | Отсутствие Bug Tracking | Хотя проект есть на GitHub, не определен процесс регистрации и отслеживания багов (GitHub Issues). Это делает управление дефектами несистемным. |

## 4. Сравнение объема долга и недоимплементированных фич

Для сравнения была проведена оценка оставшегося функционала, который не вошел в текущий прототип (MVP), но заявлен в требованиях или очевиден для будущего развития.

| Категория | Недоимплементированная фича | Оценка (человеко-часы) |
| :--- | :--- | :--- |
| **Функционал** | Интеграция с Firebase (авторизация, аналитика) | 16 |
| **Функционал** | Подключение внешнего API (Retrofit) для получения реальных данных о направлениях | 20 |
| **Функционал** | Интеграция с Google Maps SDK для "карты интересов" | 12 |
| **Функционал** | Полноценный оффлайн-режим (синхронизация данных) | 16 |
| **Дизайн** | Реализация анимаций свайпа карточек и переходов | 8 |
| **Итого** | | **72 часа** |

**Сравнительный анализ:**
*   **Объем технического долга:** ~58 человеко-часов
*   **Объем недоимплементированного функционала:** ~72 человеко-часа

**Вывод:** Объем технического долга сопоставим с объемом оставшегося функционала. Разработка новых фич на текущей "нестабильной" архитектуре приведет к многократному увеличению общего времени разработки из-за постоянного возникновения багов, сложности интеграции и высоких затрат на поддержку.

## 5. Рефлексия, выводы и корректировка плана разработки

Выполненная инспекция проекта TripSpark выявила критический уровень технического долга, который накапливался на этапе быстрого прототипирования. Основные проблемы связаны с отсутствием модульности, слоев архитектуры и систем верификации.

**Основные выводы:**
1.  **Продолжение разработки невозможно без рефакторинга:** Попытка реализовать новый функционал (API, авторизация, карты) без устранения архитектурного долга приведет к созданию нестабильного приложения и замедлит разработку в геометрической прогрессии.
2.  **Необходимость смены приоритетов:** Текущий план, ориентированный на добавление фич, должен быть пересмотрен. Приоритетом должна стать стабилизация архитектуры и процессов.

### Корректировка плана разработки проекта

**Бывший план (итеративный):**
*   Итерация 1: MVP (Текущий функционал)
*   Итерация 2: Подключение API и Firebase
*   Итерация 3: Интеграция с Google Maps

**Новый (скорректированный) план:**

| Итерация | Название | Основные задачи | Обоснование |
| :--- | :--- | :--- | :--- |
| **1.0** | **Инфраструктура и стабилизация** | - Внедрение CI/CD и Bug Tracking<br>- Создание модульной структуры<br>- Реализация слоя Domain и Repository | Создание фундамента, который позволит в будущем быстро и надежно добавлять фичи. Код становится тестируемым и поддерживаемым. |
| **1.1** | **Качество и тестирование** | - Написание Unit-тестов для критических модулей<br>- Рефакторинг статических данных<br>- Обновление документации | Обеспечение стабильности существующего MVP. Все будущие изменения будут проходить автоматическую проверку. |
| **2.0** | **Функциональное расширение** | - Подключение Firebase (авторизация)<br>- Интеграция Retrofit для получения данных<br>- Реализация оффлайн-синхронизации | На этом этапе "чистая" архитектура позволит интегрировать новые компоненты (API, БД) без переписывания UI. |
| **3.0** | **Пользовательский опыт** | - Интеграция Google Maps SDK<br>- Реализация сложной анимации<br>- Сбор аналитики | Финальный этап полировки и добавления визуально насыщенных функций, не затрагивающих ядро бизнес-логики. |

**Заключение:** Проект необходимо перевести из состояния "прототипирования" в состояние "индустриальной разработки", начав с устранения архитектурного технического долга. Рекомендуемый новый план смещает фокус с новых функций на качество и стабильность фундамента. Система после внедрения плана перейдет в состояние, пригодное для масштабирования и надежной разработки новых фич.

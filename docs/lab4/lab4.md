# Отчет по лабораторной работе №4

## Тема: Исследование стиля «Парное программирование» в проекте TripSpark

---

## 1. Какую задачу реализовывали?

Мы реализовали модуль **«Интерактивная карта интересов»**. Задача включала:

- Интеграцию Google Maps SDK в Jetpack Compose проект  
- Создание `MapViewModel` для агрегации данных из репозиториев списка желаемого (Wishlist) и рекомендаций  
- Визуализацию распределения интересов пользователя с помощью маркеров разного цвета и степени прозрачности (`alpha`), зависящей от релевантности (соответствия вектору интересов)

---

## 2. Какой стиль показался более приемлемым? Почему?

Мы протестировали два подхода:

- **Водитель–Штурман**:  
  Один разработчик писал декларативный UI на Compose, а второй параллельно проверял документацию Google Maps API и следил за правильностью внедрения зависимостей через Hilt  

- **Пинг-Понг**:  
  Один писал бизнес-логику в UseCase и ViewModel, второй — соответствующее отображение на экране карты  

**Итог:**  
Стиль **«Водитель–Штурман»** оказался более подходящим для этой задачи. Настройка Google Maps требует работы с манифестом, API-ключами и специфичными объектами (`LatLng`, `CameraPosition`), где легко допустить мелкую ошибку. Взгляд со стороны помог настроить всё с первого раза.

---

## 3. Что получилось? Положительные впечатления

- **Снижение когнитивной нагрузки:**  
  Пока «водитель» концентрировался на синтаксисе Kotlin/Compose, «штурман» продумывал логику фильтрации дубликатов между списком желаемого и рекомендациями  

- **Быстрая отладка:**  
  Ошибки с отображением маркеров (неправильные координаты) замечались и исправлялись мгновенно  

- **Обмен знаниями:**  
  Оба участника разобрались в том, как работает `maps-compose` и как прокидывать состояние карты через `rememberCameraPositionState`

---

## 4. Что не получилось? Как можно это поправить?

- При удаленной работе через Android Studio (Pair Drop / Code With Me) иногда возникали конфликты при индексации проекта после добавления новых зависимостей в `libs.versions.toml`  
- Обсуждение цветовой палитры маркеров заняло больше времени, чем само написание кода  

**Как исправить:**

- Проводить синхронизацию Gradle по очереди  
- Заранее согласовывать дизайн-систему (цвета, иконки) в Figma или отдельном документе перед началом сессии  

---

## 5. Будете ли использовать ПП для других задач? Каких?

Да, в проекте TripSpark парное программирование будет крайне полезно для:

- Рефакторинга слоя данных: выделение мапперов (Mappers) для преобразования объектов БД в доменные модели  
- Реализации алгоритмов ранжирования: настройка весов в векторе интересов пользователя  
- Настройки Firebase Cloud Messaging: сложные конфигурации пуш-уведомлений, требующие тестирования на разных устройствах
